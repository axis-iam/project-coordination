Designed from real multi-repository delivery work, without tying the workflow to a specific product or stack.

The central rule

Separate coordination, execution, and acceptance.

01

Coordinate

Create the canonical task, resolve dependencies, assign worktrees, declare the validation harness, and produce a handoff prompt.

02

Execute in a fresh session

Implement only the declared scope, run the harness, and append an execution record. Do not silently expand work or self-accept.

03

Accept independently

Return to coordination to check diff, evidence, dependencies, and consumer contracts before recording acceptance.

Scale the process

Use only the amount of process the work needs.

Compact

Local, low-risk changes can stay in one session with a focused validation command.

Standard

Backend and frontend work uses task records, contract ownership, a fresh execution session, and coordinator acceptance.

Complex

Multiple repositories, SDKs, parallel work, migrations, and external gates receive explicit dependency and release controls.

What stays local

A coordination kernel, not a product template.

Product boundaries, runtime launchers, service topology, domain onboarding, API conventions, and project-specific coding rules remain in the target repository. This skill coordinates those capabilities; it does not replace them.
